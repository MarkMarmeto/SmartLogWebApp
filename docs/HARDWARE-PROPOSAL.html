Executive Summary

We propose a dedicated on-premises server bundle for the SmartLog Web Application — the central hub that all gate scanner stations connect to. The server hosts the admin portal, ingests QR scan submissions from gate devices, manages the SQL Server database, dispatches SMS notifications to parents, and runs daily background jobs for alerts and data archival.

The bundle is built around a dedicated mini-PC (Intel Core i5, 12th Gen) with 16 GB RAM and a 512 GB NVMe SSD — sufficient to run SQL Server and the ASP.NET Core web application simultaneously under sustained load. It is paired with a 1,500 VA UPS (larger than the gate scanner UPS) to provide a clean database shutdown window during power outages, an external backup drive for weekly SQL Server backups, and a USB-to-serial adapter for the optional GSM modem SMS path.

What the Admin Server Does

The SmartLog Web Application is an ASP.NET Core 8 monolithic application that fulfills several distinct roles simultaneously. All of these run on the same server machine:

Admin Portal (Razor Pages)
Browser-based management UI for administrators, teachers, and security staff. Handles student records, QR generation, enrollment, attendance reports, SMS templates, and system configuration.
Scan Ingestion API
REST endpoint (POST /api/v1/scans) that receives QR scan submissions from all gate scanner PCs. Validates HMAC signatures, checks duplicates, records attendance, and queues parent SMS — all within a single HTTP request.
SQL Server Database
Stores all persistent data: student records, scan logs, SMS queues, audit logs, calendar events, program and section data, profile pictures, and retention archives. This is the most memory-intensive workload on the server.
SMS Worker Service
Background service that polls the SMS queue every 5 seconds, dispatches messages via GSM modem (serial port) or Semaphore (cloud API), retries failures with exponential backoff, and logs delivery outcomes.
No-Scan Alert Service
Daily background job (runs at 18:10 by default) that identifies students who did not scan in today and queues a parent notification SMS for each. Checks scanner health before sending to avoid false alerts.
Data Retention Service
Daily background job that purges old SMS logs, audit records, and scan history per configurable retention policies. Writes CSV archive files to disk before deletion for offline data recovery.
Network position in the deployment The server sits on the school's dedicated SmartLog LAN switch alongside the gate scanner PCs. It holds a static LAN IP (e.g., 192.168.10.1) on its wired Ethernet port, which scanner devices connect to for scan submission. The server's Wi-Fi or second Ethernet port connects to the school router for internet access — required for Semaphore SMS and software updates. These two network paths are kept on separate subnets.

Technical Justification

Why 16 GB RAM (not 8 GB)

The server runs two memory-intensive processes simultaneously:

  • SQL Server Express: Microsoft recommends a minimum of 4 GB RAM for SQL Server. In practice, SQL Server will consume 4–8 GB of buffer pool memory to cache frequently accessed data pages (student records, scan history, SMS queue) — this is normal and intentional behavior. Constraining SQL Server to 8 GB total means only ~4 GB remains for the OS and web application.
  • ASP.NET Core web application: Under normal load (several concurrent admin users + active scanner connections), the web application consumes 500 MB–1 GB. Peaks occur during report exports and end-of-day No-Scan Alert processing.

16 GB RAM ensures SQL Server can cache working data in memory, the web application has headroom for request bursts, and the OS has breathing room — with no risk of memory pressure causing slow query times or request drops.

Why 512 GB NVMe SSD (not 256 GB)

The server's storage is consumed by several distinct workloads:

  • OS + SQL Server + application: ~40–60 GB for Windows 11, SQL Server binaries, .NET Runtime, and the web application.
  • SQL Server database: Capped at 10 GB per database under SQL Server Express. For a school with 1,000 students generating ~5 scans/day for 200 school days/year, the scan table grows by ~3–4 million rows/year — well within the 10 GB Express limit even over 5 years.
  • Profile pictures: Student, faculty, and admin profile pictures stored on disk. At ~100 KB average compressed JPEG per student, 1,000 students = ~100 MB.
  • Retention CSV archives: Daily SMS and scan archives written before data deletion. At conservative estimates, ~50–100 MB/month of CSV data.
  • SQL Server backup files: If backups are stored locally before copying to the external drive, full weekly backups may reach 1–5 GB each.
  • Windows updates and logs: Windows Update cache and Serilog log files.

A 256 GB SSD becomes tight within 2–3 years once all the above are factored in. A 512 GB SSD provides comfortable headroom for the full 5-year deployment lifecycle.

Why a 1,500 VA UPS (not 650 VA like the gate scanners)

The admin server is the single most important machine in the deployment. Unlike a gate scanner — which can stop and lose a few scans to its local queue — the admin server losing power mid-transaction risks SQL Server database corruption:

  • SQL Server uses write-ahead logging (WAL), but an abrupt power loss can corrupt the transaction log, requiring database recovery that may fail on an uninitialized machine.
  • The SMS worker may be mid-send when power is cut; incomplete SMS queue records require manual cleanup.
  • Retention archive writes (CSV files) may be partially written, producing incomplete archive files.

A 1,500 VA / 900 W UPS provides approximately 15–25 minutes of runtime for this hardware load — enough for automatic graceful shutdown via UPS management software, without relying on an operator being present. The 650 VA units used at gate scanners would only provide 5–8 minutes, which is marginal for a database shutdown sequence. The additional cost (~₱2,000 more than a 650 VA unit) is the best-value reliability investment in this proposal.

Why a dedicated external backup drive

A mini-PC SSD is a single storage device with no redundancy. While NVMe SSDs are reliable, any storage device can fail. SQL Server databases stored only on the internal SSD represent a single point of failure for all school attendance records, student data, and SMS logs. An external USB drive provides:

  • Weekly full backups: scheduled via Windows Task Scheduler using SQL Server's native backup to .bak file.
  • Point-in-time recovery: if the internal SSD fails, the school can restore from the most recent weekly backup on a replacement machine.
  • Offline retention archives: CSV files from the retention service can be copied to the external drive for long-term offline storage beyond the server's own disk.

Why SQL Server Express (not Standard)

SQL Server Express 2022 is free with no licensing cost and supports up to 10 GB per user database. For a school with 300–1,000 students running SmartLog, the total database size is projected to remain well below 5 GB for the first 3–5 years. If a school expands significantly (multiple campuses, very large enrollment), migration to SQL Server Standard becomes an option — but at deployment time, Express eliminates software licensing cost entirely. The ASP.NET Core application and .NET 8 Runtime are open-source and also free.

Why a mini-PC (not a full tower server or NAS)

  • Appropriate scale: this is a single-school, single-tenant application with a relatively light database workload. Enterprise server hardware (rack-mount, redundant PSUs, hot-swap drives) is overspecified and significantly more expensive.
  • Low power consumption: a mini-PC with a 12th Gen Intel i5 consumes ~15–25 W at idle, compared to 100–200 W for a tower workstation. For an always-on server, this represents meaningful electricity savings over 5 years.
  • Thermal reliability: unlike a laptop, a mini-PC maintains rated CPU clock speeds indefinitely without throttling, as its thermal design targets continuous (not burst) operation.
  • Compact footprint: fits in a server closet, admin office cabinet, or on a desk shelf without requiring dedicated rack space.

Database size projections

All figures assume SQL Server Express 2022 (10 GB per database limit). Data below is conservative — actual figures depend on school size, SMS volume, and retention policy settings.

Data category Estimated size Notes
Student & enrollment records ~5–20 MB Even at 1,000 students with full enrollment history, this is negligible.
Scan records ~20–40 MB/year At 2 scans/student/day × 200 school days × 1,000 students = 400,000 rows/year. Each row ~50–100 bytes.
SMS queue & logs ~30–60 MB/year Depends on SMS volume (entry/exit opt-in rate and broadcast frequency). Purged on a configurable retention schedule.
Audit logs ~10–20 MB/year Admin actions, login history, QR regenerations. Also subject to retention purge.
QR codes (payload + image) ~100–200 MB Base64-encoded QR images stored in the database. One row per student; grows slowly (only on re-issuance).
Total projected (5-year, 1,000 students) ~500 MB–1.5 GB Well within the 10 GB Express limit. Larger schools or higher SMS volume may reach 2–4 GB. If the school approaches the limit, retention policies can be tightened to purge older records earlier.

Risk Mitigation
Risk Mitigation built into this proposal
Power outage corrupts SQL Server database 1,500 VA UPS provides 15–25 min runtime. Combined with UPS management software, the server initiates a graceful SQL Server shutdown automatically before battery depletes. No manual operator intervention required.
Internal SSD failure causes data loss Weekly full SQL Server backups to the external USB drive. Restoring a weekly backup to a replacement machine recovers all student records, scan history, and configuration. Maximum data loss is one week of new scan records (recoverable from paper log if needed).
Database reaches SQL Server Express 10 GB limit Retention policies in the admin UI control data growth. SMS logs and audit records have configurable retention windows. CSV archives of purged data are written to disk before deletion, preserving historical data in a queryable format.
Internet outage prevents Semaphore SMS delivery SmartLog supports a GSM modem as a local fallback. The SMS worker respects a per-provider routing config and can fall back to the modem automatically. The optional 4G LTE USB dongle provides internet failover independent of the school's main router.
Server is unavailable during school hours Gate scanner PCs include a local offline queue that stores accepted scans during server downtime. When the server comes back online, the scanner app automatically retransmits queued scans. No scans are lost due to a temporary server outage.
Unauthorised physical access to the server Server should be placed in a locked cabinet or server room. Windows 11 Pro includes BitLocker full-disk encryption — recommended to enable, protecting student PII on the SSD even if the machine is physically removed from the premises.
Hardware obsolescence over 5-year lifecycle All components are commodity x86 hardware. The mini-PC runs standard Windows 11 Pro — replacement units, RAM upgrades, and SSD replacements are available from any local PC retailer throughout the deployment lifecycle.

Implementation Notes

Software installation sequence

  1. Install Windows 11 Pro and apply updates.
  2. Install SQL Server Express 2022 and configure the SQL Server service to start automatically.
  3. Install .NET 8 ASP.NET Core Runtime (server hosting bundle).
  4. Deploy the SmartLog Web Application to C:\SmartLog\ and configure it as a Windows Service via sc create or the included deployment script.
  5. Set environment variables: SMARTLOG_DB_CONNECTION, SMARTLOG_HMAC_SECRET_KEY, SMARTLOG_SEED_PASSWORD.
  6. Run the application once to trigger EF Core database migrations and seed default roles.
  7. Configure Windows Firewall to allow inbound connections on port 5051 (scanner LAN) and block external access on the internet-facing adapter.
  8. Set up Windows Task Scheduler for the weekly SQL Server backup script.
  9. Configure UPS management software for automatic graceful shutdown at 20% battery remaining.

Network configuration

  • Assign static IP 192.168.10.1 / 255.255.255.0 (no gateway) on the Ethernet port connected to the SmartLog LAN switch.
  • Leave the Wi-Fi adapter on DHCP from the school router for internet access (Semaphore SMS, Windows Update).
  • If a 4G LTE dongle is included, configure it as the secondary internet adapter with a higher metric — it activates automatically only when the primary internet path is unreachable.

Procurement & lead time

  • All components are available from local distributors (PC Express, DataBlitz, Lazada/Shopee Mall).
  • Estimated procurement lead time: 1–2 weeks for stock items.
  • Server setup and software installation: approximately 4–6 hours including OS installation, SQL Server, application deployment, and network configuration.
  • Full system commissioning (server + all gate PCs connected and tested): approximately 1 day.

Warranty & lifecycle

  • Standard manufacturer warranty: mini-PC 1–3 years (brand-dependent), UPS 2 years.
  • Expected operational lifecycle: 5 years before hardware refresh.
  • Annual maintenance allowance recommended: ~5% of capital cost (~₱2,800/year) for UPS battery replacement (typically every 2–3 years), occasional peripheral replacement.
  • SQL Server Express 2022 is a current release supported by Microsoft through 2032.

Data privacy considerations (RA 10173)

The admin server stores personally identifiable information (student names, LRN, parent phone numbers, profile pictures, attendance records). Under the Philippine Data Privacy Act:

  • Enable BitLocker full-disk encryption on the server SSD.
  • Restrict physical access to the server room.
  • Limit admin portal access to authorized staff only (use the role-based access controls built into SmartLog).
  • Encrypt external backup drives containing database backups.
  • Apply retention policies to purge personal data that is no longer needed for operational purposes.
