Authentication

Scanner Device Authentication

All scanner-facing endpoints require the X-API-Key header:

X-API-Key: sk_live_xxxxxxxxxxxx

API keys are generated in Admin > Register Device. Keys are shown once and stored as SHA-256 hashes — constant-time comparison prevents timing attacks. Revoked devices are rejected immediately.

Admin/User Authentication

Dashboard, attendance, report, and profile picture endpoints use cookie-based session auth (10-hour sliding expiration). Log in at /Account/Login.

Dashboard API

All dashboard endpoints require cookie auth (CanViewAttendance policy).

{ "totalStudents": 450, "presentToday": 312, "absentToday": 138, "attendanceRate": 69.3 }

GET /api/v1/dashboard/recent-activity

QR Code Format Reference

SMARTLOG:{studentId}:{timestamp}:{HMAC-SHA256-base64}

Example:
SMARTLOG:2026-07-0001:1739512547:a7Bx9kL2mN4pQ6rS8tU0vW2xY4zA6bC8==
PartDescription
SMARTLOGFixed prefix — identifies this as a SmartLog QR code
studentIdStudent's ID (YYYY-GG-NNNN format)
timestampUnix epoch seconds at QR generation time
HMAC-SHA256-base64HMAC-SHA256(studentId:timestamp) signed with SMARTLOG_HMAC_SECRET_KEY

The scanner app validates the HMAC locally, then submits the full raw qrPayload string to POST /api/v1/scans. The server re-validates independently. The timestamp does not expire — deduplication is server-side.

Security Notes

  • Store API keys in platform secure storage — never hardcode in the scanner app.
  • Use HTTPS in production to prevent man-in-the-middle attacks.
  • Log only the first 20 characters of QR payloads to protect student data.
  • Rate limit scan submissions on the scanner side (e.g., 1 per second).
  • API key hash comparison uses CryptographicOperations.FixedTimeEquals — do not implement your own comparison.
