System Overview

SmartLog is a two-application system deployed entirely on the school's local network. No internet connection is required for core operations.

Roles & Responsibilities

🖥️ SmartLogWebApp (Server)
  • Source of truth for all student, attendance, and config data
  • Validates QR codes (HMAC-SHA256)
  • Enforces business rules: school calendar, duplicate detection, student status
  • Queues and sends SMS notifications to parents
  • Provides admin UI (Razor Pages) for managing the entire system
  • Exposes a REST API for scanner clients
📷 SmartLogScannerApp (Client)
  • Reads QR codes via camera or USB barcode scanner
  • Performs local HMAC pre-validation before hitting the server
  • Submits scans and shows color-coded results to the gate guard
  • Monitors server health every 15 seconds
  • Multiple instances can run simultaneously — each with its own API key

The Scanner App and Web App communicate exclusively over HTTP/REST on the school's LAN.

Scanner → Server Request Flow

1
Student presents QR code to the gate scanner
2
Scanner reads payload: SMARTLOG:{studentId}:{timestamp}:{hmac}
3
Scanner validates HMAC locally — rejects bad format or tampered codes immediately, without a network round-trip
4
Scanner sends POST /api/v1/scans with X-API-Key header and { qrPayload, scannedAt, scanType }
5
Server authenticates device (SHA-256 hash lookup, constant-time comparison)
6
Server re-validates HMAC independently
7
Server checks: student active? school day? duplicate within 5-minute window?
8
Server saves Scan record to SQL Server and queues SMS notification
9
Server responds: { studentName, grade, section, status: "ACCEPTED" }
10
Scanner shows color-coded result + plays audio cue
11
SmsWorkerService (background) sends SMS to parent via GSM modem or Semaphore

Authentication & Trust Model

Scanner Device → Server

Each scanner device has a unique API key (sk_live_xxx). The Web App stores only the SHA-256 hash. The Scanner stores the plain key in platform secure storage (DPAPI on Windows, Keychain on macOS).

Scanner App             Web App
X-API-Key: sk_live_xxx ──►  SHA256(key)
                             == Device.ApiKeyHash?
                             (constant-time compare)

Keys are never stored in config files, logs, or source code.

QR Code Validation (HMAC)

The HMAC secret is shared between the Web App (signs QR codes at generation) and the Scanner App (local pre-validation).

Scanner (local pre-check):
  HMAC-SHA256(id:ts) == payload.hmac?

Server (re-validates independently):
  HMAC-SHA256(id:ts) == payload.hmac?
  (constant-time compare)

Both validations use CryptographicOperations.FixedTimeEquals to prevent timing attacks.

Admin Browser → Server

Standard cookie-based auth via ASP.NET Core Identity. 10-hour sliding expiration. Role-based policies (SuperAdmin, Admin, Teacher, Security, Staff) restrict access to pages and API endpoints.

QR Code Lifecycle

Generation (Web App)

1
Admin generates QR for a student
2
Web App computes HMAC-SHA256(studentId:timestamp) using SMARTLOG_HMAC_SECRET_KEY
3
Payload: SMARTLOG:{id}:{ts}:{hmac} saved to QrCode table (IsValid=true)
4
QR image printed on student ID card

Revocation (Web App)

1
Admin regenerates QR for a student (lost card, etc.)
2
Old QrCode.IsValid set to false
3
New QR generated with a new timestamp and HMAC
4
Any future scan of the old QR → REJECTED_QR_INVALIDATED
The timestamp in the QR code does not expire — it is only used as input to the HMAC signature, not for time-based validity. Deduplication (5-minute window) is enforced server-side per scan submission.

SMS Notification Flow

Scan accepted by Web App
    │
    ▼
ScanService.QueueAttendanceNotificationAsync()
    ├─ Is SMS globally enabled?          (SmsSettings)
    ├─ Does student have a phone number? (Student.ParentPhone)
    ├─ Is SMS enabled for this student?  (Student.SmsEnabled)
    ├─ What language?                    (Student.SmsLanguage → EN or FIL)
    ├─ Render template                   (SmsTemplate: ENTRY or EXIT)
    ├─ Duplicate check (5-min window)
    └─ INSERT into SmsQueue (Status=Pending)
              │
              ▼
    SmsWorkerService  (background, polls every 5s)
    ├─ Fetch next Pending message (Priority order, then ScheduledAt)
    ├─ Try GSM_MODEM (default)
    │       ├─ Success → Status=Sent, log to SmsLog
    │       └─ Failure → try fallback if enabled
    ├─ Try SEMAPHORE (fallback)
    │       ├─ Success → Status=Sent, log to SmsLog
    │       └─ Failure → schedule retry
    └─ Retry: exponential backoff 2 min → 4 min → 8 min (max 3 retries)

Offline Resilience (Scanner App)

The Scanner App handles temporary server unavailability through a SQLite-backed offline queue.

Normal Operation

QR scan
  → HMAC local pre-validation
  → POST /api/v1/scans
  → display result to guard

Server Unreachable

QR scan
  → HMAC local pre-validation
  → POST fails
  → OfflineQueueService
  → INSERT into SQLite QueuedScans

Sync on Reconnect

BackgroundSyncService (polls health endpoint every 15s)
    │
    ├─ Server reachable?
    │     → flush QueuedScans to server in batches
    │     → success: delete from queue / mark synced
    └─ Still unreachable?
          → continue queuing, retry next cycle
Offline mode is currently disabled by default ("always-online mode"). The scanner rejects scans when the server is unreachable unless offline mode is re-enabled in code.

Key Constraints

  • The server must have a static IP — scanner devices use it as their server URL
  • All devices must be on the same LAN subnet — no internet or cross-subnet routing required
  • Each gate PC registers as a separate device with its own API key
  • The admin UI is accessible from any browser on the LAN — no dedicated admin machine required
  • The .NET 8 runtime is self-contained in the published output — no SDK required on gate PCs
