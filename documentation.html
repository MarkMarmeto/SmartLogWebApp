SmartLog Web Application

SmartLog is a School Information Management System built for Philippine K-12 schools. It tracks student attendance via QR code scanning, sends SMS notifications to parents/guardians, and provides administrative tools for managing students, faculty, academic years, calendar events, and reporting.

Target Environment
  • Philippine K-12 schools (500–2,000 students)
  • Offline-first LAN operation
  • Bilingual SMS (English / Filipino)
  • DepEd LRN compliance
Companion Application

SmartLogScannerApp — .NET MAUI desktop scanner client that reads student QR codes (camera or USB keyboard wedge) and submits scans to this web application via REST API.

Architecture

Monolithic ASP.NET Core 8 application with Razor Pages for the UI and REST API controllers for scanner device integration and AJAX-driven dashboards.

Authentication & Authorization

Uses ASP.NET Core Identity with cookie-based authentication. Session expires after 10 hours with sliding expiration.

Password Policy

  • Minimum 8 characters
  • Requires uppercase, lowercase, digit, and special character
  • Account lockout: 5 failed attempts → 15-minute lockout
  • New accounts require password change on first login (MustChangePassword flag)

Cookie Security

  • HttpOnly, Secure (HTTPS in production), SameSite=Strict
  • Cache-Control: no-cache, no-store (prevents back-button caching)

API Key Authentication (Scanner Devices)

Scanner devices authenticate via X-API-Key header. Keys are generated during device registration and stored as SHA-256 hashes in the database. Comparison uses constant-time equality to prevent timing attacks.

Key format: sk_live_ prefix + random bytes

Key rotation: Admins can regenerate API keys from the Devices page. The old key is immediately invalidated.

QR Code System

QR Payload Format

SMARTLOG:{studentId}:{unixTimestamp}:{base64HmacSha256}

Example: SMARTLOG:2026-07-0001:1739512547:5Dkd9j2xQ8kL...
  • Prefix: SMARTLOG (case-sensitive)
  • Student ID: YYYY-GG-NNNN format
  • Timestamp: Unix seconds (UTC) when QR was generated
  • Signature: HMAC-SHA256 over {studentId}:{timestamp} using shared secret
The HMAC secret is set via the SMARTLOG_HMAC_SECRET_KEY environment variable. The scanner app does NOT need this secret — it simply reads and forwards the QR payload to the web app for server-side verification.

Attendance Processing Flow

Scanner Device SmartLog Web Server | | | POST /api/v1/scans | | X-API-Key: sk_live_xxx | | {qrPayload, scannedAt, scanType} | |--------------------------------------->| | |-- Authenticate device (API key hash) | |-- Parse QR payload (4 colon-separated parts) | |-- Verify HMAC-SHA256 (constant-time) | |-- Lookup student (must be active) | |-- Check calendar (must be school day) | |-- Check duplicate (5-min window) | |-- Save Scan record | |-- Queue SMS notification (async) | {scanId, studentName, grade, | | section, status: ACCEPTED} | |<---------------------------------------|

SMS Notification System

Dual Gateway Architecture

GSM Modem (Primary)
  • Serial port AT commands (COM3, 9600 baud)
  • Offline-capable (no internet needed)
  • Rate limiting: 3s delay between messages
  • Circuit breaker: 3 failures → 5-min pause
Semaphore API (Fallback)
  • Cloud HTTP API (api.semaphore.co)
  • Requires internet connectivity
  • Returns ProviderMessageId for tracking
  • Webhook for delivery receipts

Queue Processing

  • SmsWorkerService (BackgroundService) polls every 5 seconds
  • Fetches pending messages ordered by Priority, then CreatedAt
  • Respects ScheduledAt for delayed delivery
  • Retry with exponential backoff: 2, 4, 8 minutes (max 3 retries)
  • On success: Status=Sent, logged to SmsLog with ProviderMessageId

Bilingual Templates

Templates support English and Filipino with placeholders: {StudentName}, {Grade}, {Section}, {Time}, {Date}. Language is per-student (SmsLanguage field).

School Calendar

Event Types

TypeEffect on Attendance
HolidaySchool closed — scans rejected as REJECTED_NOT_SCHOOL_DAY
EventSchool activity — may or may not affect attendance (configurable)
SuspensionClass suspension — can target specific grade levels via AffectedGrades JSON array

Calendar events integrate with scan processing: ICalendarService.IsSchoolDayAsync(date, gradeLevel) is called during scan validation.

Enrollment & Batch Re-enrollment

  • Students have one active enrollment per academic year (Student → Section → GradeLevel)
  • Annual batch re-enrollment: Admin selects source → target year, marks each student as Promote / Graduate / Skip
  • Executes atomically: deactivates old enrollments, creates new ones in a single transaction
  • Supports CSV bulk import for initial student data loading

Device Management

  • Register scanner devices at /Admin/RegisterDevice — generates API key shown once
  • Device list at /Admin/Devices with status, last-seen timestamp, and actions
  • Revoke: Immediately prevents device from submitting scans
  • Reactivate: Restores a revoked device
  • Regenerate Key: Issues new API key, old key stops working immediately
  • All device actions are audit-logged

Security Features

  • API key hashing: SHA-256, never stored in plain text
  • Constant-time comparison: Prevents timing attacks on API key and HMAC validation
  • CSRF protection: Anti-forgery tokens on all forms (ASP.NET Core default)
  • Security headers: X-Frame-Options: DENY, CSP, XSS protection
  • Password enforcement: Complexity requirements + mandatory change on first login
  • Audit logging: All admin actions logged with user, IP, and timestamp
  • Sensitive data: HMAC secret and DB credentials via environment variables only
  • SMS secrets: Never exposed in rejection messages or logs
  • Account lockout: 5 failed attempts → 15-minute lockout
  • Cache control: No browser caching of authenticated pages
